1 Executive Summary

This report characterizes the Telugu OCR corpus assembled by a teammate from Telugu Wikisource (Telugu Wikisource contributors 2024) and publicly released as AlbertoChestnut/telugu-ocr on Hugging Face (Chestnut 2026). The corpus comprises 217 books and ~32,949 aligned page pairs (scan image + human transcription) totaling approximately 10.3 GB. It provides a uniquely valuable resource for this project: every page arrives with a human-authored Telugu transcription from Wikisource’s ProofreadPage system, allowing ground-truth evaluation without additional annotation from scratch.

Note

Corpus selection: The course’s corpus guidance [Week 1 announcement] identifies several acceptable public Telugu OCR sources, including Wikisource-derived collections. This Wikisource-sourced corpus was selected as the project’s primary dataset on that basis. Supplementary datasets (e.g. AI4Bharat’s Indic OCR resources (AI4Bharat 2024)) may be incorporated in later phases for additional evaluation diversity.

2 Corpus Overview

2.1 Source and Provenance

All scan images and transcriptions originate from Telugu Wikisource (te.wikisource.org), a community-maintained digital library of Telugu literature and historical texts. The curation pipeline:

  1. Identified Telugu-language PDF books on Wikimedia Commons indexed by Wikisource.
  2. Filtered to books with ≥ 80% Telugu script characters (measured by Unicode block ఀ–౿), ≥ 50 scan pages, and at least one transcribed page.
  3. Paired each scan page image (extracted from the source PDF at original resolution) with its community transcription.
  4. Applied Wikisource’s ProofreadPage quality labels to every page.

The resulting dataset is licensed CC BY-SA 4.0, permitting unrestricted academic use.

3.3 Telugu Script Ratio

The original dataset curation pipeline applied a ≥ 80% Telugu Unicode block (ఀ–౿) threshold when selecting books for inclusion, per the dataset’s published methodology. However, the per-book Telugu ratio value itself was not included in the downloaded meta.json metadata, so it cannot be independently re-verified or visualized from the files available to us.

Note

Known limitation: telugu_ratio is reported as N/A in our generated statistics because the source metadata does not expose this field. If exact per-book ratios are needed for Phase 2 filtering decisions, they would need to be computed directly from the transcription text (counting Unicode code points in the Telugu block) rather than read from existing metadata. This is a candidate follow-up task, not a blocker — the curator’s 80% inclusion threshold already provides a reasonable assurance that script contamination is low across the corpus.

Note

On DPI specifically: pixel dimensions alone don’t determine DPI — DPI depends on the physical page size the scan represents, which isn’t recorded in this corpus’s metadata. As a rough proxy, assuming a typical book page (~6×9 in), the measured 1,500px width corresponds to roughly 250 DPI — close to, though slightly under, the assignment’s 300 DPI minimum.

A more notable finding than the DPI estimate itself: width is exactly 1,500px across all 56 sampled images — mean, median, min, and max are identical, zero variance. Height varies normally (1,284–3,395px), as expected from real page-to-page aspect ratio differences. Width showing literally no variation across an independently random sample is not something natural scanning produces — it strongly suggests the corpus curator (or Wikisource’s PDF-to-JPEG export pipeline) standardized every page to a fixed 1,500px width before publishing, leaving height to vary with each page’s actual proportions. This is a real, useful finding for preprocessing design: every image in this corpus can be assumed to already share a consistent horizontal scale, with no separate per-page width normalization step required.

This wasn’t pursued further given the project’s actual OCR pipeline (Claude + Tesseract) showed no systematic resolution-driven failures in practice — see Phase 2/3 findings in the final report.

5 Telugu Script Challenges

Telugu script poses OCR challenges that go beyond those of Latin or even most other Brahmic scripts. The following issues are particularly salient in this corpus.

5.1 Conjunct Characters

Telugu consonants combine into ligatures (కూటాక్షరాలు) whose shape differs substantially from either component character. For instance, the cluster k+ṣ (క + ష) produces the conjunct క్ష, a visually distinct glyph. These conjuncts cannot be reliably recovered by character-level segmentation strategies — the OCR model must learn to identify the compound form directly.

Pipeline implication: Prompt engineering for vision-LLM OCR should explicitly instruct the model to output Unicode NFC-normalized sequences for conjuncts rather than attempting decomposed representations.

5.2 Vowel Diacritics (Mātrās)

Telugu vowel diacritics (ఆ-కారాలు / mātrās) attach to consonants in spatially complex ways — appearing above, below, before, or after the base glyph, and sometimes spanning multiple positions. Examples:

  • కా (ka + ā-mātrā, appended right)
  • కి (ka + i-mātrā, prepended left)
  • కు (ka + u-mātrā, below right)
  • కూ (ka + ū-mātrā, below, extending further right)

Confusion among mātrās is the single most common OCR error in Telugu systems (Rao et al. 2019). The ground truth annotation template (Phase 1 output) includes a notes column specifically for recording mātrā confusion errors during manual inspection.

5.3 Font Diversity

Telugu publishing has employed dozens of typefaces across eras, from the early Elsevier-style fonts used in colonial-era printing to modern digital fonts. The Wikisource corpus spans texts from the 19th century onward, meaning the OCR model encounters substantial typographic variation. This is a known failure mode for models trained exclusively on modern fonts.

5.4 Low-Resource Context

Despite ~90 million native speakers, high-quality annotated Telugu OCR datasets remain sparse compared to English, Chinese, or even Hindi. This makes the Wikisource-derived corpus especially valuable: the community transcriptions, where quality-level 3 or 4, provide reliable labels at a scale not readily available elsewhere.

6 Ground Truth Sampling Strategy

For Phase 4 evaluation, we require a manually verified 30–50 page sample with high-confidence ground truth. We use the following stratified sampling design:

Code 
gt = pd.DataFrame({
    "Quality Level": ["4 — validated", "3 — human_proofread", "1–2 — lower quality", "0 — no_text"],
    "Target Share": ["40%", "30%", "20%", "10%"],
    "Target Count (n=40)": [16, 12, 8, 4],
    "Rationale": [
        "Gold-standard labels; primary benchmark set",
        "Reliable; expands evaluation coverage",
        "Stress-tests pipeline on noisy inputs",
        "Edge-case handling (blank / image-only pages)",
    ],
})
gt.set_index("Quality Level")

The achieved sample differs slightly from this target: the corpus contains no quality-0 (“no_text”) pages, so that allocation was redistributed, and the sampler’s top-up logic shifted some quality-3 slots to fill the gap, yielding 16 validated / 16 human_proofread / 4 problematic / 4 not_proofread pages (see Annotation Results below).

Pages were selected using 3_sample_ground_truth.py with a fixed random seed (42) for reproducibility. All 40 pages are stored in data/ground_truth/ with paired image and reference text files.

Manual annotation protocol: For each sampled page, an annotator:

  1. Opened the .jpg file and the paired .txt (Wikisource transcription).
  2. Compared the transcription against the actual page image.
  3. Recorded Confirmed in manual_transcription if the transcription matched, or supplied a correction if it did not.
  4. Noted artifact types and assigned a subjective scan quality score (1–5).

Annotation was performed jointly using direct visual inspection and an LLM-assisted comparison workflow (image + transcription pairs reviewed turn-by-turn) to cross-check character-level accuracy, given the density of Telugu conjuncts and diacritics.

6.1 Annotation Results

Code 
gt_actual = pd.DataFrame({
    "Quality Level": ["4 — validated", "3 — human_proofread", "2 — problematic", "1 — not_proofread"],
    "Achieved Count": [16, 16, 4, 4],
    "Confirmed Accurate": [16, 16, 3, 4],
    "No Transcription Available": [0, 0, 1, 0],
})
gt_actual.set_index("Quality Level")

Key finding: Of the 40 sampled pages, 39 had an existing Wikisource transcription that was confirmed accurate against the scanned image with no corrections needed — a strong validation that quality levels 1, 3, and 4 are usable as-is for Phase 4 ground truth. One page (quality level 2, “problematic”) had no transcription text at all in the source file, despite carrying page-level metadata. This is a meaningful finding: not every page labeled “problematic” merely contains errors — some contain no transcription whatsoever. This supports excluding quality level 2 from the Phase 4 benchmark set entirely, rather than treating it as “noisy but usable.”

No corrections were required for any of the 39 transcribed pages in this sample, which is a notably higher accuracy rate than initially expected even for quality levels 1 and 2. This may reflect that Wikisource contributors who initially transcribe a page (level 1) are often reasonably accurate even before formal proofreading; the “not_proofread” label reflects review status, not necessarily poor transcription quality. This is worth re-testing with a larger sample in later phases.

7 Preprocessing Design Implications

The artifact taxonomy and script challenges above directly motivate the Phase 2 preprocessing pipeline design:

Challenge Phase 2 Response
Ink fading (observed in 18/40 sampled pages) Contrast enhancement (CLAHE) + upscaling
Smudging / blur (observed in 9 and 7 of 40 pages respectively) Adaptive binarization; RealESRGAN for severe cases
Low-resolution scans (pending full corpus measurement) Bicubic upscaling to ≥ 300 DPI equivalent before model input
Non-text regions (illustrations/diagrams, 5/40 pages) LayoutParser region detection; exclude from text OCR pass
Page skew / binding shadow (not yet quantified) Deskewing as a universal first step; re-evaluate need after Phase 2 image sampling
Mātrā / conjunct confusion Unicode NFC normalization on all model outputs; prompt instructs model to preserve diacritics
